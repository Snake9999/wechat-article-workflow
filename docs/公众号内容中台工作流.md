     1|# 公众号内容中台工作流（WeWe - RSS 上游 Feed 中台版）
     2|
     3|> 目标：把WeWe - RSS 作为上游公众号 Feed 中台，负责统一订阅、历史同步、增量抓取与全文输出；下游再完成清洗、改写、md2wechat 渲染，并自动推送到公众号草稿箱。
     4|
     5|## 1. 设计目标
     6|
     7|这个工作流不是把“抓取、改写、发布”硬写死在一个脚本里，而是拆成 5 层：
     8|
     9|1. 上游 Feed 中台层：WeWe - RSS
    10|2. 采集编排层：拉取新增文章、去重、清洗、入库
    11|3. 内容加工层：摘要提取、双版本改写
    12|4. 微信发布层：md2wechat 渲染 + 微信草稿箱 API
    13|5. 运营控制层：人工审核、飞书通知、失败重试、审计日志
    14|
    15|这样做的好处：
    16|- 公众号信源接入和内容生产解耦
    17|- 上游可替换，下游不需要重写
    18|- 历史文章资产掌握在自己手里
    19|- 可以先自动入草稿箱，再人工审核发布
    20|- 故障容易定位，不会一挂全挂
    21|
    22|## 2. 已确认的关键事实
    23|
    24|基于公开资料和代码结构，已确认：
    25|
    26|1. WeWe - RSS 适合作为“公众号 Feed 中台”
    27|- 支持微信公众号订阅
    28|- 支持历史文章同步
    29|- 支持后台定时更新
    30|- 支持 RSS / Atom / JSON 输出
    31|- 支持全文输出模式
    32|
    33|2. 历史数据不是必须存在第三方云端
    34|- 自建部署时，历史数据保存在你自己的数据库中
    35|- 支持 MySQL（推荐）
    36|- 支持 SQLite（可用，但官方不推荐）
    37|- Prisma schema 中明确存在 Feed / Article / Account 表
    38|
    39|3. md2wechat 适合作为下游渲染与草稿上传组件
    40|- Markdown / HTML 可转公众号图文格式
    41|- 可上传封面图到微信素材库
    42|- 可调用微信公众号 draft/add 创建草稿
    43|

   173|## 7. 模块职责拆分
   174|
   175|### 7.1 信源配置中心
   176|
   177|维护一份公众号 source registry，建议字段：
   178|- source_id
   179|- source_name
   180|- category
   181|- priority
   182|- enabled
   183|- need_history_sync
   184|- sync_frequency
   185|- include_keywords
   186|- exclude_keywords
   187|- owner
   188|- note
   189|
   190|作用：
   191|- 控制抓哪些号
   192|- 按分类路由到不同下游策略
   193|- 决定是否抓历史文章
   194|- 决定优先级和更新频率
   195|
   196|### 7.2 WeWe - RSS 上游中台
   197|
   198|建议职责：
   199|- 统一接入所有公众号
   200|- 管理抓取账号 / 订阅源
   201|- 同步历史文章
   202|- 按 cron 增量更新
   203|- 对外提供统一 feed/API
   204|- 保留基础抓取存档
   205|
   206|建议部署：
   207|- 生产环境优先 MySQL
   208|- 测试环境可用 SQLite
   209|- 独立容器部署
   210|- 不与改写服务混部署
   211|
   212|### 7.3 采集编排 Worker
   213|
   214|建议职责：
   215|- 定时轮询 WeWe - RSS feed/API
   216|- 只拉最近窗口内新增文章
   217|- 标准化字段
   218|- 去重
   219|- 清洗正文

   496|## 14. 关键设计原则
   497|
   498|1. 上游采集和下游生产必须解耦
   499|2. 一定保留原文快照，避免上游内容变化导致不可复现
   500|3. 不只按 URL 去重，要加标题归一化和内容哈希
   501|4. 自动入草稿箱可以，自动群发先不要做
   502|5. 所有文章都必须保留原始来源链接
   503|6. 数据真实性优先，抓不到就如实失败，绝不模拟
   504|
   505|## 15. 风险与注意事项
   506|
   507|### 15.1 抓取源稳定性
   508|- 公众号抓取本身有上游变化风险
   509|- 需要预留重试和熔断机制
   510|- 需要监控源失效情况
   511|
   512|### 15.2 正文清洗不稳定
   513|- 公众号尾部模板、推荐阅读、广告区块经常变化
   514|- 清洗规则要可配置，不要硬编码死
   515|
   516|### 15.3 微信草稿接口限制
   517|- 标题、摘要、内容长度都有限制
   518|- 图片需先转素材并拿 media_id
   519|- 封面图尺寸、比例要符合规则
   520|
   521|### 15.4 审核合规
   522|- 自动改写必须保留原始来源记录
   523|- 需要支持人工终审
   524|- 不建议直接全自动发布
   525|
